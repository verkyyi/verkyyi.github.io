# Proposed Change: watcher-daily-digest

## Problem
Watcher generates ~16 all-clear health check entries per day in agent_log.md, each 400-600 characters. This accounts for ~7KB/day of agent_log growth — roughly 25% of the total ~29KB/day agent_log growth rate. With agent_log at 169KB and growing at ~29KB/day, it hits the 256KB tooling limit in ~1.9 days, requiring repeated emergency truncation.

PR #53 (watcher-silent-clear mode, merged 04-09) was intended to reduce watcher noise but the current format still logs every all-clear check verbosely.

## Proposed Solution
Modify watcher.yml to implement daily digest mode:
- **On state change** (failure, recovery, corrective action): log full detail entry as today
- **On all-clear**: increment a counter, do NOT write individual agent_log entry
- **Once per day** (first run after 00:00Z): write single daily summary line with count of all-clear checks, any notable metrics changes

Expected format:
```
TIMESTAMP | watcher.yml | daily-digest | 16/16 all-clear checks (prev 24h). System stable. Token: 389 pts. Logs: agent 169KB, research 45KB.
```

## Impact
- Reduces agent_log growth by ~7KB/day (25% reduction)
- Watcher still logs all failures and corrective actions in full detail
- Combined with other growth (evolve ~15KB/day, research ~17KB/day), extends time-to-limit from ~1.9 days to ~2.5 days
- Does NOT fix root cause (hourly cron frequency) but buys time

## Risk
- LOW: watcher health data preserved in daily digest
- Corrective actions and failures still logged individually
- No behavioral change to watcher health checking itself

- Alternative: issue→coder direct edit (proven pathway, avoids phantom PR risk)
