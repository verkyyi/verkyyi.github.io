# Proposed Change: Watcher Silent-Clear Mode

## Summary
Modify watcher.yml to only append agent_log entries when there is a state
change (new failure, recovery, corrective action taken). All-clear health
checks with no state change would be logged as a single counter increment
in project_state.md instead of a full entry in agent_log.md.

## Problem
Watcher all-clear entries are the dominant source of agent_log growth:
- ~24 health checks/day, each ~500-700 chars = ~15KB/day
- agent_log is 571KB (2.23x the 256KB tooling limit)
- All-clear entries contain no actionable information
- 100% of watcher checks this week were all-clear (0 corrective actions)
- Log compaction PRs (#35, #38, #41) already merged — further inline
  reduction not possible without changing what gets logged

## Proposed Implementation
In `.github/workflows/watcher.yml`, wrap the agent_log append in a
conditional: only write when the health check detects a state change
(failure, recovery, corrective action, or new PR/issue state). When
all-clear with no changes, increment a counter in project_state.md
(e.g., "consecutive all-clear checks: N") instead.

## Expected Impact
- ~80% reduction in agent_log growth (from ~4.9KB/day to ~1KB/day)
- No loss of actionable information — failures and actions still logged
- project_state.md health section already captures watcher status
- Slows the log crisis: buys ~4 months additional runway at reduced rate

## Risk
- LOW: Failure entries are unaffected — only suppresses all-clear noise
- Reversible: can restore full logging by removing the conditional
