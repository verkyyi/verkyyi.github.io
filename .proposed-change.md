# Proposed Change: Auto-rebase for merge-conflict-blocked PRs

## Problem
PRs approved by reviewer get stuck on merge conflicts caused by high-frequency
state file commits (~4/hour). PR #4 has been blocked 21+ hours, PR #5 for 6+
hours. The watcher detects these but can only label `needs-human` — it cannot
resolve them. This is the #1 systemic bottleneck in the pipeline.

## Root Cause
Every evolve and watcher run commits state file updates to main. PRs created by
coder branch off main but diverge within 1-2 hours due to state/ file changes.
By the time reviewer approves, the branch has conflicts in state/ files.

## Proposed Solution
Add a rebase step to the coder or reviewer workflow: when a PR is approved but
has merge conflicts limited to `state/` files, auto-rebase the branch onto main
using a "theirs" strategy for state files (main's state is always more current).

### Implementation sketch (workflow YAML addition)
```yaml
# In reviewer.yml or a new auto-rebase.yml
- name: Auto-rebase on state conflicts
  if: contains(github.event.pull_request.labels.*.name, 'auto-merge')
  run: |
    git fetch origin main
    git checkout ${{ github.head_ref }}
    git rebase origin/main -X theirs -- state/
    git push --force-with-lease
```

## Alternatives Considered
1. **Reduce state commit frequency** — would help but degrades state visibility
2. **Exclude state/ from PRs** — breaks the "every agent action is a commit" rule
3. **Human rebase** — current approach, doesn't scale

## Risk
- Force-push to PR branches (mitigated by --force-with-lease)
- Could overwrite intentional state changes in PRs (mitigated by limiting to
  state/ files only and using main's version which is always more recent)

## Autonomy Tier
PR with `needs-review` — this is a workflow YAML change.

## Source
Weekly analysis 2026-03-23 — pattern: 2/2 approved PRs blocked on state/ merge
conflicts for 6-21+ hours.
