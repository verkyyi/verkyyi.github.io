# Proposed Change: Reduce evolve.yml and watcher.yml cron frequency

## Problem
State commits account for 96% of all git activity (~48 commits/day from evolve + watcher running hourly). This causes merge conflicts on every PR branch within hours of creation, creating a human-dependent deadlock that has blocked all forward progress for 4 consecutive weeks. PR #4 (landing page) has been stuck for 117+ hours solely due to state-commit-induced merge conflicts.

## Proposed Change
Reduce cron schedules from hourly to every 3 hours:
- `evolve.yml`: change `cron: '0 * * * *'` to `cron: '0 */3 * * *'` (8 runs/day instead of 24)
- `watcher.yml`: change cron similarly (8 runs/day instead of 24)

Expected result: ~16 state commits/day instead of ~48, reducing merge conflict window by 67%.

## Files to modify
- `.github/workflows/evolve.yml` — line 5, cron schedule
- `.github/workflows/watcher.yml` — cron schedule (same change)

## Evidence
- Week 3 analysis: 353 commits, 339 state (96%), merge conflicts on all 4 PRs
- Week 4 analysis: 376 commits, 361 state (96%), same 4 PRs still blocked
- No human activity detected that would benefit from hourly monitoring
- Research sources (tokenman, quarto-cli) update at most daily — hourly polling yields identical results 23/24 times
- OpenAI blog source was polled 50+ times before removal, all blocked — proving hourly is wasteful
- Pipeline health checks find same "all clear" status 90%+ of runs

## Risk
- Lower frequency means slower detection of pipeline failures (3h max vs 1h)
- Mitigated by: workflow_dispatch still available for manual triggers, and failures that need immediate attention are rare (4 total in 5 days of operation, all resolved within one cron cycle)
