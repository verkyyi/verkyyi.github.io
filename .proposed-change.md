# Proposed Change: Inline Log Truncation in Analyze Workflow

## Problem
agent_log.md is 420KB+ (exceeds the 256KB tooling read limit). research_log.md
is 172KB/1243 lines and approaching the same limit. Three separate PRs for log
archival (#39, #42, #48) have been approved by the reviewer but ALL are
merge-blocked by conflicts from hourly state commits. This is the same systemic
pattern that has blocked cron fixes for 15 weeks. The log reduction campaign
(PRs #35/#38/#40/#41) reduced *growth rate* by ~60-70%, but existing bloat
remains and continues accumulating. Analysis tooling is now degraded — the
analyze workflow cannot fully read its own logs.

## Proposed Solution
Add an inline log truncation step to analyze.yml that runs before the weekly
analysis Claude CLI call. When agent_log.md exceeds 200KB, move all entries
older than 14 days to `state/archive/agent_log_YYYY-MM.md` (append if exists).
When research_log.md exceeds 100KB, apply the same treatment. This is a state
file operation and qualifies as AUTO tier per CLAUDE.md autonomy rules.

## Implementation
In `.github/workflows/analyze.yml`, add a step before the Claude CLI invocation:

```yaml
- name: Truncate oversized logs
  run: |
    mkdir -p state/archive
    CUTOFF=$(date -u -d '14 days ago' +%Y-%m-%d)
    
    # Archive agent_log entries older than 14 days
    if [ $(stat -c%s state/agent_log.md 2>/dev/null || echo 0) -gt 204800 ]; then
      MONTH=$(date -u -d '14 days ago' +%Y-%m)
      # Keep header (lines 1-3) and recent entries, archive the rest
      head -3 state/agent_log.md > /tmp/agent_header.md
      awk -v cutoff="$CUTOFF" '
        NR<=3 {next}
        /^[0-9]{4}-[0-9]{2}-[0-9]{2}/ {
          date=substr($0,1,10)
          if (date < cutoff) {print >> "/tmp/agent_archive.md"; next}
        }
        {print}
      ' state/agent_log.md > /tmp/agent_recent.md
      cat /tmp/agent_header.md /tmp/agent_recent.md > state/agent_log.md
      if [ -f /tmp/agent_archive.md ]; then
        cat /tmp/agent_archive.md >> "state/archive/agent_log_${MONTH}.md"
      fi
    fi
    
    # Archive research_log entries older than 14 days  
    if [ $(stat -c%s state/research_log.md 2>/dev/null || echo 0) -gt 102400 ]; then
      MONTH=$(date -u -d '14 days ago' +%Y-%m)
      head -3 state/research_log.md > /tmp/research_header.md
      awk -v cutoff="$CUTOFF" '
        NR<=3 {next}
        /^[0-9]{4}-[0-9]{2}-[0-9]{2}/ {
          date=substr($0,1,10)
          if (date < cutoff) {print >> "/tmp/research_archive.md"; next}
        }
        {print}
      ' state/research_log.md > /tmp/research_recent.md
      cat /tmp/research_header.md /tmp/research_recent.md > state/research_log.md
      if [ -f /tmp/research_archive.md ]; then
        cat /tmp/research_archive.md >> "state/archive/research_log_${MONTH}.md"
      fi
    fi

- name: Commit archived logs
  if: success()
  run: |
    git add state/agent_log.md state/research_log.md state/archive/ 2>/dev/null || true
    git diff --cached --quiet || git commit -m "state: archive oversized logs — analyze weekly truncation"
    git push || true
```

## Risk Assessment
- **Low risk**: Only moves old entries to archive files, never deletes data
- **AUTO tier**: State file operation per CLAUDE.md rules
- **Addresses P0**: Restores analyze tooling ability to read its own logs
- **Breaks deadlock**: Unlike PRs #39/#42/#48, this runs inline and cannot be merge-blocked
- **Idempotent**: Safe to run repeatedly — only triggers when size threshold exceeded

## Why now
The log reduction campaign has optimized growth rate. But at ~98 state commits/day,
agent_log.md will exceed 500KB within a week even with compaction. The analyze
workflow literally cannot read the file it needs to analyze. Three PRs have tried
to fix this and all hit the same merge-conflict wall. Inline truncation is the
only remaining automated path.
