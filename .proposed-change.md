# Proposed Change: Auto-Rebase for Approved Merge-Blocked PRs

## Problem
Merge conflicts are the #1 systemic issue — 16th consecutive week unresolved.
The hourly cron produces ~98 state commits/day on main, which makes every PR
branch stale within minutes. PRs #39, #42, and #48 were ALL approved by the
reviewer agent but cannot merge due to conflicts. PR #4 (landing page) has been
blocked for 625+ hours. The pipeline can autonomously detect bugs, fix them,
review fixes, and merge — but it cannot self-heal merge conflicts, which is the
single biggest bottleneck to system autonomy.

## Proposed Solution
Add an auto-rebase step to the watcher workflow. When watcher detects a PR that
is approved (reviewer-approved label or approval comment) but blocked by merge
conflicts, it rebases the PR branch onto main and force-pushes. This closes the
gap between "approved" and "merged" for PRs that are only blocked by staleness.

## Implementation
In `.github/workflows/watcher.yml`, add a step after the health check Claude CLI
call:

```yaml
- name: Auto-rebase approved merge-blocked PRs
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  run: |
    # Find PRs labeled needs-human that have reviewer approval
    APPROVED_PRS=$(gh pr list --state open --json number,labels,reviews \
      --jq '[.[] | select(.reviews[]?.state == "APPROVED" or 
        (.labels | map(.name) | any(contains("auto-merge"))))] | .[].number')
    
    for PR_NUM in $APPROVED_PRS; do
      # Check if PR has merge conflicts
      MERGEABLE=$(gh pr view $PR_NUM --json mergeable --jq '.mergeable')
      if [ "$MERGEABLE" = "CONFLICTING" ]; then
        BRANCH=$(gh pr view $PR_NUM --json headRefName --jq '.headRefName')
        echo "Rebasing PR #$PR_NUM (branch: $BRANCH)"
        
        git fetch origin "$BRANCH"
        git checkout "$BRANCH"
        
        # Attempt rebase — if conflicts are in state/ files, accept main's version
        if git rebase origin/main --strategy-option=theirs; then
          git push origin "$BRANCH" --force-with-lease
          echo "PR #$PR_NUM rebased successfully"
        else
          git rebase --abort
          echo "PR #$PR_NUM rebase failed — manual intervention required"
        fi
        
        git checkout main
      fi
    done
```

## Risk Assessment
- **Medium risk**: Force-pushes to PR branches (not main). Uses `--force-with-lease` for safety.
- **Mitigated**: Only targets PRs that are already approved. Uses `--strategy-option=theirs` for state/ conflicts (state files are always regenerated).
- **needs-review tier**: Workflow YAML change — requires human review per CLAUDE.md autonomy rules.
- **High impact**: Would unblock PRs #39, #42, #48 immediately. Addresses the #1 systemic bottleneck.
- **Structural**: This is the missing capability that prevents the pipeline from being fully autonomous for approved changes.

## Why now
The system has reached an autonomy plateau. 9 PRs merged this week (project
record), but all remaining improvements are blocked by merge conflicts. The
pipeline can detect, fix, review, and approve — but the last mile (merge) fails
because of stale branches. 3 approved PRs (#39, #42, #48) are stuck right now.
Auto-rebase is the single highest-leverage improvement for system autonomy.

## Alternatives considered
1. **Reduce cron frequency first** — Would reduce conflict rate but requires the same human YAML edit that's been blocked 16 weeks. Circular dependency.
2. **Squash-merge instead of rebase** — GitHub API merge with squash could bypass conflicts for state-only changes, but lacks the granularity of rebase for mixed PRs.
3. **Direct commits instead of PRs** — Already used for AUTO-tier. Expanding AUTO-tier would bypass review, which conflicts with safety goals.
