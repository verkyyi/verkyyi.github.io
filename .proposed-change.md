# Proposed Change: Apply cron frequency reduction to workflow YAMLs

## Problem
PR #15 (merged 2026-03-26) updated evolve_config.md with reduced cron frequency, but the actual workflow YAML files were never changed. evolve.yml still runs `0 * * * *` (hourly) and watcher.yml still runs `30 * * * *` (hourly). This produces ~48 state commits/day (96.8% of git history), causing merge conflicts on every PR branch.

Two PRs attempted to fix this:
- PR #19: reviewer failed twice, escalated to needs-human
- PR #22: closed by reviewer (rejected)

This is the **5th consecutive week** merge conflicts have been identified as the #1 systemic bottleneck. The root cause is still unfixed.

## Proposed Fix
Change cron schedules in workflow YAML files:

**evolve.yml** (line 5):
```yaml
# Before
- cron: '0 * * * *'
# After
- cron: '0 */3 * * *'
```

**watcher.yml** (line 5):
```yaml
# Before
- cron: '30 * * * *'  # Every hour at :30 (offset from evolve at :00)
# After
- cron: '30 */3 * * *'  # Every 3 hours at :30 (offset from evolve at :00)
```

## Expected Impact
- State commits reduced from ~48/day to ~16/day (67% reduction)
- Merge conflict probability drops proportionally
- PR branches can survive longer without diverging from main
- No loss of monitoring coverage — 3h interval still catches failures within same day

## Risk Assessment
- LOW risk: evolve and watcher both write state summaries; less frequent runs mean each run covers a larger time window but the same ground
- Watcher health checks every 3h is sufficient — most failures are detected within 1 run
- Growth, analyze, and deploy crons are already at appropriate frequencies and should not change
