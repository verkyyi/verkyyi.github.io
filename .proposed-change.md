## Problem

The `.proposed-change.md` workflow is broken for implementing changes:

1. Analyze/evolve writes `.proposed-change.md` describing a change
2. Coder agent creates a PR branch — but only adds/deletes the `.proposed-change.md` file itself
3. Reviewer approves and merges the file operation
4. The actual change described in the file is never executed
5. `.proposed-change.md` is consumed (deleted) with no effect

**Evidence**: PRs #54, #55, #56 (evolve_config fix) and #59 (log truncation) all merged without
making their intended change. Only PR #58 worked — because the coder agent was responding to an
issue (#57), not processing a `.proposed-change.md` file.

## Proposed Fix

For **AUTO-tier state file changes** (per CLAUDE.md autonomy rules: agent_log.md, research_log.md,
project_state.md, FEATURE_STATUS.md), the workflow that identifies the need should perform the
change directly as a state commit instead of writing `.proposed-change.md`.

### Specifically for evolve.yml and analyze.yml:

When a state file change is needed:
- **Current (broken)**: Write `.proposed-change.md` → wait for coder → phantom PR → no change
- **Proposed (direct)**: Perform the state file edit directly → commit with `state: [summary]`

This is already permitted by CLAUDE.md: "AUTO — commit directly, no PR needed: State file updates"

### Immediate application: Log truncation

The analyze workflow should directly truncate logs during its run:
```bash
# agent_log.md: keep header + last 200 entries
head -2 state/agent_log.md > /tmp/agent_log_truncated.md
tail -200 state/agent_log.md >> /tmp/agent_log_truncated.md
mv /tmp/agent_log_truncated.md state/agent_log.md

# research_log.md: keep header + last 200 entries  
head -3 state/research_log.md > /tmp/research_log_truncated.md
tail -200 state/research_log.md >> /tmp/research_log_truncated.md
mv /tmp/research_log_truncated.md state/research_log.md
```

### For non-AUTO changes

The `.proposed-change.md` pathway should still be used for PR-tier changes, but the coder
workflow needs a fix to actually implement the described change rather than just filing the
description file. Recommended: open an issue describing the phantom PR pattern so the
coder agent can investigate and fix the workflow code.

## Expected Outcome

- AUTO-tier state changes execute immediately instead of entering a broken pipeline
- Log truncation unblocked (4 failed attempts via PR pathway)
- agent_log.md: ~200 entries, ~130KB (within 256KB limit)
- research_log.md: ~200 entries, ~30KB (within 256KB limit)
- Reduces ~100 state commits/day overhead from log growth

## Risk

- Workflow behavior change requires review (needs-review tier)
- Does not fix the underlying coder agent bug for non-AUTO changes
- Log truncation deletes historical entries (acceptable: git history preserves them)
