# Proposed Change: Enable watcher to auto-close redundant PRs

## Problem
The watcher workflow detects redundant PRs (where the fix has already been merged via a different PR) and labels them `needs-human`, but cannot close them. This leads to PR queue pollution — currently 3 redundant PRs (#5, #11, #16) are accumulating, creating noise and blocking the human's ability to see which PRs actually need attention.

## Evidence
- PR #5 (growth.yml file guards) — redundant since PR #7 merged 2026-03-23
- PR #11 (analyze.yml branch collision) — redundant since PR #13 merged 2026-03-25
- PR #16 (cron frequency reduction) — redundant since PR #15 merged 2026-03-26
- All three detected as redundant by watcher, all labeled needs-human, none closeable
- 5th consecutive week with accumulating dead PRs

## Proposed Fix
In `.github/workflows/watcher.yml`, add logic for the watcher to close PRs that it has identified as redundant (where the original fix PR has already been merged). The watcher should:
1. Check if a PR is labeled `needs-human` AND the watcher's analysis identifies it as redundant
2. Add a comment explaining why the PR is being closed (e.g., "Closing: fix already merged via PR #X")
3. Close the PR via GitHub API

## Impact
- Eliminates PR queue pollution (3 dead PRs currently, will grow)
- Reduces human cognitive load when reviewing PR queue
- Prevents watcher from repeatedly re-analyzing already-dead PRs
- Low risk: only closes PRs already identified as redundant, adds explanatory comment

## Risk
LOW — Only acts on PRs the watcher has already confirmed are redundant (fix merged via different PR). Does not touch PRs with merge conflicts or active review. Closing a PR is reversible (can be reopened).

## Autonomy Tier
PR — needs-review (workflow YAML change)
