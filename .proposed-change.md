# Proposed Change: watcher-allclear-compression

## Problem
Agent_log regrows past the 25K-token Read limit within ~2 days of each truncation.
Root cause: watcher all-clear health check entries average 682 bytes each.
At ~46 watcher entries/day, this adds ~31KB/day to agent_log.
These all-clear entries carry minimal unique information — they confirm system stability
but repeat the same workflow timestamps and PR/issue counts verbatim.

## Change
In watcher.yml, compress the agent_log entry for all-clear health checks (0 corrective actions)
to a single short line:

**Before** (~682 bytes):
```
TIMESTAMP | watcher.yml | health check | all clear — 0 corrective actions. All workflows
operational (Self-Evolve TIMESTAMP, pages-build TIMESTAMP, Watcher TIMESTAMP, Weekly Analysis
TIMESTAMP). 0 failed runs in last 6h. 0 broken chains. 0 stuck runs. 2 open issues (#2, #24)
both triaged, terminal needs-human. 11 PRs needs-human (all merge-blocked, 40+ day backlog).
No recently closed issues unblock open items. Token utilization healthy (363 pts, opus-4-6,
0 max-turns, 0 rate-limit, 0 fallbacks). System stable.
```

**After** (~80 bytes):
```
TIMESTAMP | watcher | all clear | 0 actions, 0 failures, 11 PRs, 2 issues, stable
```

Only expand to full detail when corrective actions > 0 or failures detected.

## Impact
- Reduces agent_log growth from ~31KB/day to ~4KB/day (87% reduction)
- Extends time between required truncations from ~2 days to ~15 days
- No information loss: full detail still logged on state changes (failures, actions, recoveries)
- Watcher all-clear entries account for ~75% of agent_log volume

## Risk
LOW — only changes log format for no-action runs. All corrective actions, failures,
and state changes continue to use full-detail format. No behavioral change.

## Tier
PR — needs-review (workflow YAML change)
