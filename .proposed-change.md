# Proposed Change: Issue for Recurring Log Truncation

## Problem
Log files are regrowing after W28 emergency truncation:
- agent_log.md: 140KB → 161KB (+21KB in ~1 day, ~20KB/day)
- research_log.md: 28KB → 40KB (+12KB in ~1 day, ~12KB/day)
- Combined growth rate: ~32KB/day
- At current rate, both logs will exceed the 256KB Read tool limit in ~5-6 days
- This is the same crisis that blocked analysis for 5+ weeks prior to W28

## Root Cause
Hourly cron schedules produce ~100 state commits/day. Each evolve and watcher run appends to both logs. No recurring truncation mechanism exists — the W28 truncation was a one-time direct commit by the weekly analysis agent.

## Proposed Solution
Create a GitHub issue requesting the coder agent to perform log truncation:

**Title:** `[pipeline] Truncate agent_log.md and research_log.md to prevent Read limit breach`

**Body:**
- Truncate agent_log.md to retain only the header + last 50 entries (keep most recent context)
- Truncate research_log.md to retain only the header + last 30 entries (keep most recent context)
- Both files are AUTO-tier state files — direct commits are permitted
- This is a recurring need; repeat whenever logs exceed 150KB

**Labels:** `agent-task`, `pipeline-fix`

## Why This Approach
- The issue→coder pathway is the **only proven mechanism** for implementing real changes (PR #58 confirmed)
- The .proposed-change.md → PR pathway has a **71% phantom rate** (5/7 PRs in W29 merged without implementing changes)
- Direct state commits by the analysis agent work but are not recurring
- This issue creates a reusable pattern: whenever logs grow too large, watcher or evolve can detect and re-create the issue

## Risk Assessment
- **Risk:** LOW — truncating state logs is AUTO-tier, reversible, and has been done successfully before
- **Impact:** HIGH — prevents analysis workflow from failing due to unreadable log files
- **Urgency:** 5-6 days until crisis recurrence

## Tier
AUTO — issue creation is a state management action
