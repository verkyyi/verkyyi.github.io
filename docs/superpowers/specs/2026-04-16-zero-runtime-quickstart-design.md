# Zero-Local-Runtime Quickstart

## Summary

Eliminate all local runtime requirements from the quickstart. Users fork, add markdown files, set secrets, push. GitHub Actions handles parsing, adaptation, and deployment. No local Python, Node, or JSON authoring.

## Motivation

The current quickstart requires local Python (to run `adapt_all`), local Node (to preview), and hand-authoring JSON Resume format. This is too much friction for a portfolio tool. The entire pipeline can run on GitHub Actions.

## User Flow

```
1. Fork this repo
2. Replace data/resume.md with your resume
3. Add target JDs in data/jd/ (one .md per role, filename = URL slug)
4. Set secrets: GH_ISSUES_PAT, ANTHROPIC_API_KEY
5. Push — portfolio deploys to GitHub Pages
```

No JSON editing. No local runtime. Four things to do.

## Data Directory Structure

```
data/
├── resume.md           # User's resume (any format — paste from LinkedIn, PDF-to-text, etc.)
├── jd/                 # Target positions (one markdown file per role)
│   ├── cohere.md       # → serves at /c/cohere
│   └── openai.md       # → serves at /c/openai
├── slugs.json          # Auto-generated by Action from jd/ filenames
└── adapted/            # Auto-generated by Action
    ├── default.json    # Generic adaptation (no JD context)
    ├── cohere.json     # Company-specific adaptation
    └── openai.json
```

### `data/resume.md`

Any format. User pastes their resume — LinkedIn export, PDF-to-text dump, hand-written markdown, plain text. The LLM parses it into structured data during adaptation.

### `data/jd/*.md`

One markdown file per target position. Filename (without extension) becomes the URL slug. Content is freeform — paste the JD, write the role title, include a link, whatever the user has. The LLM extracts company name, role, and requirements during adaptation.

### `data/slugs.json`

Auto-generated by the Action from filenames in `data/jd/`. The frontend still reads this file at runtime (no frontend changes needed). Users never edit it.

Generated format:
```json
{
  "cohere": { "company": "Cohere", "role": "FDE, Agentic Platform", "created": "2026-04-16", "context": "auto-generated" }
}
```

The `company` and `role` values are extracted by the LLM from the JD markdown content.

### `data/adapted/*.json`

Auto-generated. One per JD plus a `default.json` (no company context). Format is JSON Resume with agentfolio meta — same as today. The frontend reads these directly, unchanged.

## Pipeline (GitHub Action)

### Trigger

Push to `main` that touches `data/resume.md` or `data/jd/*.md`.

### Process

```
For each data/jd/{slug}.md:
  Read data/resume.md + data/jd/{slug}.md
  → Single LLM call
  → Output: data/adapted/{slug}.json
    (JSON Resume with summary, highlights, section order,
     match score, skill emphasis — all generated in one call)

Also:
  Read data/resume.md (alone)
  → Single LLM call
  → Output: data/adapted/default.json

Generate data/slugs.json from:
  - Filenames in data/jd/
  - Metadata extracted during adaptation (company name, role)

Commit all generated files
Trigger deploy
```

### Single LLM Call Per Adaptation

The current multi-step pipeline (parse → score → adapt → polish) is replaced by a single LLM call per adaptation. The prompt provides the raw resume markdown + JD markdown and asks for a complete JSON Resume output with:

- Parsed and structured resume data (basics, work, projects, skills, education, volunteer)
- Tailored summary for the target role
- Reordered sections by relevance
- Rewritten/reordered highlights emphasizing relevant experience
- Match score (overall + by category)
- Matched/missing keywords
- Skill emphasis list

For `default.json`, the same call runs without JD context — just structures the resume with a generic summary.

### Match Scoring

Moved from deterministic Python keyword matching to LLM-generated scores. The LLM can reason about relevance ("3 years of Kafka experience is highly relevant to a data platform role") rather than binary keyword matching. Scores become more meaningful.

Trade-off: non-deterministic (same input might score 82% one run, 79% next). Acceptable for a portfolio site.

## What Changes

| Before | After |
|--------|-------|
| User authors `data/resume.json` (JSON Resume) | User authors `data/resume.md` (any format) |
| User authors `data/companies/*.json` | User authors `data/jd/*.md` (any format) |
| User edits `data/slugs.json` manually | `slugs.json` auto-generated from `data/jd/` filenames |
| Local `python -m scripts.adapt_all` | GitHub Action runs on push |
| Multi-step pipeline (parse → score → adapt → polish) | Single LLM call per adaptation |
| Deterministic match scoring in Python | LLM-generated scores |
| `resume.json` is source of truth | `resume.md` is source of truth |
| `chat_answer.py` reads `resume.json` | `chat_answer.py` reads `resume.md` |
| `data/companies/` directory | `data/jd/` directory |
| Multiple slugs per company supported | One file = one slug (filename is the slug) |

## What Stays the Same

- Frontend (React SPA) — still reads `data/adapted/*.json` and `data/slugs.json`
- Deploy workflow — still builds and deploys to Pages
- Chat widget — still works (reads `resume.md` instead of `resume.json`)
- Analytics — unchanged
- URL routing — `/c/{slug}` pattern unchanged

## Scripts Changes

### New: `scripts/adapt_from_markdown.py`

Single-call adaptation script. Takes `resume.md` + optional `jd/*.md` → outputs adapted JSON. Replaces `adapt_one.py` and `adapt_all.py` for the Action pipeline.

### Modified: `scripts/chat_answer.py`

`build_system_prompt` reads `data/resume.md` instead of `data/resume.json`.

### Kept (for contributors): existing scripts

`adapt_one.py`, `adapt_all.py`, etc. remain for local development but are not part of the quickstart.

## Workflow Changes

### Modified: `.github/workflows/adapt.yml`

Triggers on push to `data/resume.md` or `data/jd/*.md`. Runs the new `adapt_from_markdown.py`, generates `slugs.json` + `adapted/*.json`, commits results, triggers deploy.

## Sample Data
